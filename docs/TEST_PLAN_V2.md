# ValueLog v2 E2E 수동 테스트 플랜

이 문서는 ValueLog v2 시스템의 핵심 워크플로와 제약 사항(S6)을 포함한 수동 E2E 테스트 시나리오입니다. 
배포/릴리스 전 이 테스트 플랜의 모든 항목을 통과해야 합니다.

## 시나리오 1: 핵심 기록 흐름 유지 여부 (퀘스트 기반)
1. **가입/로그인**: 신규 혹은 기존 계정으로 로그인합니다.
2. **온보딩 통과(신규 계정인 경우)**: 목표 설정 등 온보딩 절차가 완료되는지 확인합니다.
3. **퀘스트 진행**: 홈 화면의 "오늘의 퀘스트 완료" 버튼을 클릭합니다.
   - 카테고리(예: '창작/취미')를 선택합니다.
   - 모든 필수 질문에 텍스트를 입력하고 끝까지 진행합니다.
   - 사진 추가 단계에서는 "사진 없이 다음" 또는 이미지를 첨부합니다.
4. **결과 확인**:
   - 퀘스트 완료 화면에 XP와 "증빙 링크 / 결과 신호 / 역량 체크" 버튼 목록이 보이는지 확인.
   - 경험이 DB `experiences` 테이블에 정상적으로 INSERT 되는지 점검합니다.

## 시나리오 2: 기록 강화(Enrichment) 및 Trust 변화
1. **증빙(Evidence) 링크 추가**: 
   - 퀘스트 완료 혹은 경험 상세 화면에서 `🔗 증빙 링크` 탭을 엽니다.
   - `https://github.com` 등 유효한 링크를 입력하고 저장합니다.
2. **성과(Impact Signal) 추가**: 
   - `📊 성과 수치` 탭에서 "결과물 생성 완료" 등 텍스트를 입력하고 저장합니다.
3. **역량(Competency) 체크**:
   - `💡 역량 체크` 탭에서 "문제해결"을 선택하고 3개의 항목을 체크합니다.
4. **결과 확인 (단일 소스 오브 트루스 적용 검증)**:
   - "저장하고 완료"를 누릅니다.
   - 홈을 지나 타임라인(내 기록 보기)에서 방금 생성한 경험 상세 페이지로 진입합니다.
   - 상단 뱃지가 **"Evidence"** 라벨 (파란색)로 올라가 있고 점수가 부여되었는지 확인합니다 (`trust_score` >= 30).
   - `🌿 성장 L3.0` 과 같은 Growth Index 배지가 표시되는지 확인합니다.
   - DB에 `trust_label`, `growth_index` 가 계산된 값으로 저장되었는지 점검합니다.

## 시나리오 3: 룰베이스 Stats 지표의 정합성
1. 기록 여러 건을 추가합니다. (예: 5건. 그 중 2건은 증빙 링크 첨부).
2. 하단 내비게이션의 `📊 Stats` 탭에 진입합니다.
3. **지표 검증**:
   - `증거 포함률`이 정확히 40% (2/5건)인지 확인.
   - 역량을 추가한 다른 경험이 있다면 `역량 커버리지`가 n/8 으로 정상 반영되었는지 확인.
   - 하단의 **"인사이트"** 박스에 룰베이스 기반 팁(예: "오늘은 증빙 링크 1개만 붙여보세요")이 출력되는지 확인.

## 시나리오 4: Row Level Security (RLS) 무결성
1. **계정 A**로 로그인하여 '개인적인 비밀 경험'을 하나 기록합니다. 
   - 이 경험의 ID (`experience_id`)를 따로 복사/기억해둡니다.
2. 로그아웃 후 **계정 B**로 로그인합니다.
3. 화면상 타임라인에서 계정 A의 경험이 전혀 보이지 않는지 확인합니다 (노출 0).
4. **API 직접 호출 테스트 (Optional but Recommended)**:
   - 계정 B의 브라우저 콘솔에서 계정 A의 경험 ID로 `supabase.from('experiences').select().eq('id', '계정A의_EXP_ID')` 를 호출하여 반환값이 `[]` 인지 확인합니다.

## 시나리오 5: 스킬과 역량의 분리 적용
1. 경험 상세 화면에 진입합니다.
2. `🛠️ 스킬 태그` 섹션이 보이는지 확인하고 "추가" 버튼을 누릅니다.
3. 드롭다운에서 `"React"` 또는 `"Python"` 등을 선택 후 저장합니다.
4. 역량(Competency)은 객관식 체크리스트(Lv.1~5)로 계산되고, 스킬(Skill)은 단순 태깅 형식으로 확연히 분리되어 동작하는 것을 확인합니다.

## 시나리오 6: 프로젝트 및 Peer 리뷰 (기능 플래그 OFF/ON 확인)
1. 현재 애플리케이션의 기본 설정상 **Peer Review** 및 **Project Mode**로 진입할 수 있는 UI 버튼이 숨겨져 있어야 합니다. (기능 플래그 OFF 상태 점검)
2. (플래그 ON 테스트 - 개발용): 코드 상에서 기능 플래그 변수를 `true`로 켰을 때, 특정 모드 진입이 일어나는지, 관련된 라벨(Peer, Issuer)이 활성화 가능한지 제한적으로 확인합니다.
