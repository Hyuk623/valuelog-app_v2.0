# 주요 기술 의사결정 내역 (ValueLog V2)

### D-0007 활용도 점수(Quality Score) 기반 유도형 UX 모델 채택
- 날짜: 2026-04-07
- 상황: 사용자가 기록을 단순히 '기록'하는 데 그치지 않고, 나중에 취업이나 포트폴리오에 쓸 수 있도록 '자산' 수준으로 품질을 높이게 할 동기가 필요했음.
- 결정: 신뢰성(Trust)과는 별도의 '완성도 점수(Quality Score)' 체계를 도입. 100점 만점으로 계산하되, 부족한 영역을 부드러운 가이드(Tip) 형태로 노출하여 강제성 없이 자발적 고도화를 유도함.
- 이유: 강제적인 '기록 다시 쓰기'나 '저장 차단'은 사용자 경험을 해치고 기록을 귀찮게 만듦. 게임화 요소와 코칭 메시지를 결합한 '조용하지만 강력한 유도(Nudge)' 방식이 제품 철학에 부합함.
- 영향 범위: 경험 저장 로직, 타임라인 및 상세 UI, 전체 통계 요약.

### D-0006 기록 일관성 및 신뢰성을 위한 투명한 Audit Log (투명성 모델) 채택
- 날짜: 2026-04-04
- 상황: 시스템 내 인증(Peer/Issuer Review 등)을 받은 후에도 자유롭게 기록 텍스트를 수정할 수 있으면, 외부인이 열람할 때 신뢰도가 하락한다는 제품 기능적 문제 발생 (위변조 우려).
- 결정: 기존 버전들의 내용을 JSON 스냅샷 형태로 DB(`experience_edit_logs`)에 보관하고, 상세 뷰 화면에서 모든 열람자가 "기존 변경 기록 스냅샷"을 직관적으로 확인할 수 있도록 기능(수정내역 모달)을 제공함.
- 이유: 기록을 아예 수정하지 못하게 엄격하게 막으면 사용자의 기록 편의성이 크게 저하됨. '투명한 이력 공개'로 타인의 자율적인 검증과 신뢰를 이끌어내는 모델(옵션 B)이 ValueLog의 취지와 더 맞음. 
- 대안: 수정 발생 시 기존의 모든 검증 점수(Trust Score)를 삭제 또는 무효화함 (기각됨 - 아주 사소한 오탈자 수정에도 큰 패널티가 가해짐).
- 영향 범위: `experiences` 업데이트 라이플사이클 전반, 데이터베이스 용량, 상세 조회 API.



### D-0005 다이얼로그 모달 컴포넌트 계층 구조(Z-Index & Portal) 패턴 확립
- 날짜: 2026-04-04
- 상황: 전역 레이아웃 컴포넌트(`BottomNav` 등)가 내부 라우트 컴포넌트의 다이얼로그 박스 및 저장 컨테이너를 가리는 레이아웃 깨짐 현상 발생.
- 결정: 향후 화면(모바일 뷰 등) 하단 또는 중앙에 올라오는 중요한 알럿(삭제 모달, 약관 모달 등)은 모두 `createPortal`을 이용하여 `document.body` 레벨로 끌어올린 후 z-index를 최상단(9999 등)을 주어 렌더링하도록 컨벤션화. 개별 페이지에 귀속된 하단 고정 버튼(예: 저장하기 버튼)은 전역 랩퍼(50)보다 높은 클래스(z-[60])를 부여해 덮어쓰도록 처리.
- 이유: CSS 스태킹 컨텍스트(Stacking Context) 충돌 방지 및 레이어 안전성 확보.
- 대안: 전역 상태 중앙 관리(Redux, Zustand 모달 관리자) - 추후 복잡해질 시 이관 고려. 본 변경건에서는 UI 렌더링 계층만 조정.
- 영향 범위: `ExperienceDetailPage`, `ExperienceEditPage` 및 향후 모든 다이얼로그 모달 UI.


### D-0004 재가입 시 온보딩 플로우 강제화
- 날짜: 2026-03-13
- 상황: 계정 삭제 후 동일 계정(이메일 등)으로 다시 가입했을 때 기존 데이터가 남아있거나 온보딩(초기 설정)을 건너뛰는 문제 발생.
- 결정: 기존 사용자 데이터를 강제 삭제 처리하고 프로필이 없는 사용자는 즉각 온보딩(닉네임, 튜토리얼) 페이지로 라우팅되도록 라우터 및 인증 로직 개선.
- 이유: 일관된 User Experience 보장 및 남아있는 가비지 데이터 방지.
- 대안: 소프트 삭제. 본 시스템은 데이터 완전 영구 삭제 채택.
- 영향 범위: 전체 회원 탈퇴 로직 및 인증 라우팅 상태.

### D-0003 Firebase Hosting 환경 배포 채택
- 날짜: 2026-03-10
- 상황: 개발이 완료된 리액트 코드의 스테이징 및 프로덕션 호스팅 환경이 필요했음.
- 결정: 현재 프로젝트 구조와 호환성이 좋은 Firebase Hosting을 사용하여 정적 배포 설정.
- 이유: 기존 백엔드가 Supabase이긴 하나 프론트 배포가 가볍고 간단하게 이뤄지는 환경이 선호됨.
- 대안: Vercel, Netlify.
- 영향 범위: 빌드 아티팩트(`dist/`) 생성 과정 및 CLI 배포 절차.

### D-0002 ESLint 의존성 배열 엄격 규제 적용
- 날짜: 2026-02-28
- 상황: 컴포넌트 렌더링 시 Hook 사용 규칙 미준수로 Vite 프로덕션 빌드가 중단됨(Exit 1).
- 결정: `eslint-disable`을 남발하지 않고 `useEffect` 뎁스 배열 및 지역 변수를 철저히 수정 적용 리팩토링.
- 이유: 빌드 안정성 뿐만 아니라, 이후 무한 렌더링 등의 심각한 런타임 발생 예방.
- 대안: ESLint 빌드 에러 무시 설정 도입 (기각).
- 영향 범위: 프론트엔드 React 컴포넌트.

### D-0001 Trust & Growth 지표 시스템 (calcTrust)
- 날짜: 2026-02-26
- 상황: 기존의 단순 경험치 누적 모델을 넘어 평가자와 동료 피드백을 수용하는 '신뢰도' 점수 필요.
- 결정: `calcConfidence` 함수를 폐기하고 리뷰 데이터(Peer/Issuer)를 합산하여 `trust_score`와 `trust_label`을 부여하는 `calcTrust` 기반 모델로 대규모 개편.
- 이유: 외부 인증을 더 가치 있게 취급하여 시스템 전반의 품질을 제고함.
- 대안: 평가가 반영되지 않는 자율 기입 방식 유지 (기각).
- 영향 범위: `competencies.ts`, `types/index.ts` 및 관련 UI UI.
